\chapter*{Заключение} \label{ch-conclusion}
\addcontentsline{toc}{chapter}{Заключение}

В данной работе разработано консольное приложение на JavaScript для анализа палиндромных подстрок с использованием алгоритма Манакера.
Поставленная цель достигнута: спроектированы сценарии использования, реализованы два конкурирующих алгоритма решения задачи, сформированы структуры входных и выходных данных, а также создан пользовательский консольный интерфейс с загрузкой из JSON и сохранением результата в файл.

В части проектирования тестов были использованы четыре техники: разбиение на классы эквивалентности, анализ граничных значений, statement testing и branch testing.
Для них были составлены проверяемые условия, базовые и оптимизированные контрольные примеры, а затем эти сценарии были перенесены в реальный тестовый набор проекта.

Для модульного тестирования выбраны Jest, Sinon и Chai.
Преимущества такого набора состоят в простоте запуска, хорошей совместимости с Node.js, наличии параметризованных тестов, выразительных assertion и полноценной поддержке test doubles.
К недостаткам можно отнести отсутствие встроенного assumption-style механизма и необходимость комбинировать несколько библиотек вместо одного монолитного решения.
Этот недостаток был компенсирован введением собственного assumption-style helper-модуля.

Итоговый тестовый набор включает 7 test suite и 48 тестов.
В нём присутствуют позитивные и негативные сценарии, параметризованные проверки, не менее пяти способов assertion, два assumption-style метода, три разновидности test doubles и сравнение основного алгоритма с эталонным конкурирующим решением.
По отчёту Jest получено 98.07\% line coverage по проекту и 100\% line coverage для вычислительных модулей.

Мутационное тестирование с помощью StrykerJS показало следующие итоговые значения: LCC = 98.07\%, MSI = 71.03\%, MCC = 100\%, CoveredCodeMSI = 71.03\%.
После доработки тестов результат улучшился по сравнению с первоначальным прогоном, а модуль \texttt{resultWriter.js} достиг 100\% MSI.
Полностью mutation-adequate набор пока не является, однако качество тестов можно считать высоким и существенно улучшенным по итогам мутационного анализа.

Использование TDD в контексте данной работы оказалось оправданным.
Его преимуществами стали ранняя фиксация ожидаемого поведения, удобная декомпозиция задачи на небольшие проверяемые шаги и быстрое обнаружение регрессий.
Основной недостаток TDD проявился в высоких трудозатратах на формализацию тестов для алгоритмических деталей и на последующее усиление тестов по результатам мутационного тестирования.
Тем не менее, именно такой подход позволил получить проект, который соответствует заданию как по содержанию отчёта, так и по реальному коду приложения и тестов.
