\chapter{Описание экспериментов по фаззингу} \label{ch2}

Данная глава содержит результаты запусков фаззера, входные данные, позволившие
выявить ошибки, а также оценку времени, памяти, удобства и эффективности
динамического анализа.

\section{Условия проведения эксперимента} \label{ch2:sec1}

Эксперименты выполнялись в локальном каталоге приложения. Для каждого fuzz target
использовалась команда с ограничением \texttt{-runs=20000}. Фактически оба дефекта
были найдены до достижения установленного лимита запусков.

Перед запуском фаззинга выполнялась проверка обычных unit-тестов командой
\texttt{npm test}. Это нужно для отделения уже известных регрессионных ошибок от
дефектов, найденных фаззером. Unit-тесты проекта завершились успешно: 7 test suite
и 64 теста прошли. Следовательно, найденные фаззером ошибки относятся не к
основным проверенным сценариям алгоритма Манакера, а к дополнительным модулям
обработки входных строк.

Для каждого запуска фиксировались следующие признаки:
\begin{itemize}

    \item стек вызовов;

    \item количество запусков до обнаружения ошибки;
    \item объём памяти, если он был выведен фаззером до аварийного завершения.
\end{itemize}

В обоих случаях ошибка была найдена практически сразу. Для \texttt{datasetNameFuzzTarget}

Base64-представлением \texttt{JQ==}.

В первом эксперименте фаззер завершился уже на первом запуске, потому что пустая
строка является базовым элементом начального корпуса. Это означает, что дефект
является грубым: функция не защищена даже от минимального входа. Во втором
эксперименте фаззеру понадобилось 134 запуска, чтобы получить символ
\texttt{\%}. Это также быстрый результат, так как вход состоит всего из одного
байта и сразу приводит к \texttt{URIError}.

\section{Обсуждение результатов} \label{ch2:sec4}

Jazzer.js показал высокую скорость поиска ошибок в выбранных модулях. Оба дефекта
были найдены без ручного подбора входных данных. Детализация сообщения достаточна
для локализации ошибки: вывод содержит тип исключения, стек вызовов, строку файла
и минимальный вход, сохранённый в crash-файл.

Удобство фаззера оценивается как высокое по следующим критериям:
\begin{itemize}
    \item минимальная настройка для Node.js-проекта;
    \item возможность запуска через npm-скрипты;
    \item сохранение crash-файлов;
    \item вывод минимального входа;
    \item понятный стек вызовов.
\end{itemize}

Ограничение фаззера состоит в том, что он сам не объясняет бизнес-смысл ошибки.
Например, сообщение \texttt{URIError: URI malformed} указывает на место сбоя, но
разработчик должен самостоятельно определить, что вход должен валидироваться как
пользовательская строка и не должен аварийно завершать приложение.

Для модуля \texttt{datasetNameParser.js} фаззер показал, что функция не проверяет
структуру входной строки перед обращением к элементу массива. Для модуля
\texttt{percentInputDecoder.js} фаззер показал, что стандартная функция
\texttt{decodeURIComponent} может аварийно завершить обработку некорректного
пользовательского ввода.

Сравнение с ручным тестированием показывает, что фаззинг полезен именно для
неожиданных входов. Ручные тесты обычно проверяют корректные имена датасетов и
несколько очевидных ошибок. Фаззер же сразу проверяет пустой ввод, одиночные
символы, бинарные последовательности и некорректные комбинации байтов.

\section{Выводы} \label{ch2:conclusion}

Фаззинг оказался удобным и эффективным для поиска ошибок в функциях, работающих
с пользовательским вводом. Для данного приложения он особенно полезен на границах
форматов: имена датасетов, JSON-строки, percent-кодирование и строковые параметры
алгоритма.
